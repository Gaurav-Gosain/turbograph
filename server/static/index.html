how turbograph worksclose

turbograph answers questions grounded in your own documents. You bring the documents and a local model; it does the retrieval.

the pipeline

Documents are split into chunks, each chunk is embedded into a vector, and similar chunks are linked into a graph. A question is embedded too, the closest chunks are found, and the graph spreads relevance to neighbours that are related but not a direct match. The model then answers from those chunks.

the graph

Each dot is a chunk. Colour is its community, a cluster of closely related chunks. Lines connect similar chunks. Hover a dot to preview it, click to inspect it and its neighbours, drag to rearrange, scroll to zoom.

chat and citations

An answer streams in and lists the chunks it used as chips. Hover a chip to light up its dot in the graph, click it to jump there. The graph search box does the same lookup without asking the model.

settings

results is how many chunks to ground on. graph blend moves between pure similarity (0) and pure graph propagation (1). diversity trades some relevance for less repetitive results.

buckets

Each bucket is a separate corpus with its own graph, so you can keep different document sets apart. Re-uploading a changed document updates it in place and only re-embeds the parts that changed.

keyboard

Ctrl K command palette, / search the graph, ? commands, esc close or stop a running answer, enter send, shift enter newline.

the .tg store formatclose
a .tg file is one complete corpus: its chunks, their embeddings, the content history, and an optional entity graph, in a single portable file. save one, copy it anywhere, load it, and you are back exactly where you left off without re-embedding anything. each bucket is one .tg file.

technical

the file is a single encoding/gob stream of one snapshot struct — no magic number, no header, no text framing. gob is self-describing, so a file written by an older build loads in a newer one and fields added later read as their zero value when absent. that is the whole compatibility story; there is no version integer to bump. 
type snapshot struct {
    Cfg       Config                  // how this corpus was built
    Dim       int                     // embedding dimension
    Chunks    []Chunk                 // the text, in ingestion order
    Embeds    [][]float32             // raw embeddings, one per chunk
    Hashes    map[string][32]byte     // doc id -> content hash
    Entities  []entity.Entity         // entity graph nodes (optional)
    Relations []entity.Relation       // entity graph edges (optional)
    Versions  map[string][]docVersion // per-document content history
}
only the inputs that are expensive to recompute are stored; everything derived is rebuilt deterministically on load. that keeps the file small, immune to index-internal layout changes, and forward-compatible. a bring-your-own chunker is code, not data, so it is not persisted — reattach it after loading if you ingest more documents with it. full spec in docs/format.md.
